Dus, wat is een LLM? Hoe worden ze gemaakt?

  • Het klassieke proces van het maken van een A.I. Model is eenvoudig:
    • Je hebt data nodig
    • Label die data
    • Train een model op die data
    • Gebruik het model om voorspellingen te doen op vergelijkbare gevallen
  • Voorbeeld: Om een risico model te maken dat faillissementen van bedrijven voorspelt:
    • Verzamel informatie over bestaande bedrijven (omzet, kosten, grootte, etc.)
    • Label welke bedrijven failliet gingen en welke niet
    • Train een model op deze dataset
    • Gebruik het model om risico scores te produceren voor toekomstige bedrijven
  • Dit heet supervised training. Hoe meer data je hebt, hoe beter je model presteert.
  • Gelabelde data krijgen is duur:
    • Als het 5 cent kost om Ã©Ã©n record te labelen, kosten 1 miljoen records $50,000
    • Dubbel-checken van elk record zou $100,000 kosten
  • Een LLM is een specifiek model getraind op natuurlijke taal bronnen, wat unsupervised training mogelijk maakt.
  • Je hoeft de data niet te labelen; labels worden afgeleid van de trainingsdata zelf.
  • Tekst is overal, waardoor het makkelijk is om enorme datasets te verzamelen voor training.
  • Met toegang tot enorme data (zoals het hele publieke internet) en de middelen voor unsupervised training, kunnen we nu LLM's maken.
  • ChatGPT, bijvoorbeeld, is een LLM getraind op een diverse range van internet tekst.

Based on Probability

Tuning Parameters
  • Dus wat gebeurt er onder de motorkap? De meeste modellen die we vandaag gebruiken heten "next token generators." Gegeven een input, gebruiken ze probabilistische berekeningen om een output te genereren.
  • Wat betekent dat? Bijvoorbeeld, als ik je vraag wat je favoriete wereldkeuken is, zou je kunnen antwoorden "Vietnamese." Als ik je een minuut later weer vraag, geef je waarschijnlijk hetzelfde antwoord. Een model echter, zou een ander antwoord kunnen geven de tweede keer, omdat zijn responses gebaseerd zijn op waarschijnlijkheid, niet zekerheid.
  • De aard van deze modellen is probabilistisch, niet deterministisch. Daarom is het onmogelijk om 100% voorspelbare antwoorden te garanderen. Er zijn technieken om consistentie te verbeteren, maar het is belangrijk om te begrijpen dat deze technologie fundamenteel gebaseerd is op waarschijnlijkheidsberekeningen.
  • Als de modellen altijd compleet willekeurig waren, zouden ze nutteloos zijn. Gelukkig zijn er parameters die je kunt aanpassen, zoals "top-p" en "temperature," om de willekeurigheid te controleren. (Demo deze instellingen.)
  • DEMO: Dit wordt duidelijker als we naar een klein voorbeeld kijken. De temperatuur past de waarschijnlijkheidsverdeling aan via een wiskundige functie. Bij een lage temperatuur wordt het verschil tussen de waarschijnlijkheden van woorden groter, waardoor woorden met een hoge waarschijnlijkheid vaker gekozen worden. Bij een hoge temperatuur wordt de verdeling gelijkmatiger, waardoor ook woorden met een lagere waarschijnlijkheid meer kans krijgen om geselecteerd te worden.
  • Top-P werkt als een filter of een venster. Bij een hoge waarde is het venster groot en mogen meer woorden meedoen om als volgend woord gekozen te worden. Bij een lage waarde is het venster smal en komen minder woorden in aanmerking.

LLMs excel where both the input and output are words, and the rules are fuzzy.

  • Text generation and completion
  • Language translation
  • Code generation and debugging
  • Question answering and reasoning
  • Creative writing and content creation
  • Data analysis and summarization

  • Maar waar zijn ze goed in? En waar zijn ze niet goed in? Laten we dat even verkennenâ€¦
  • LLMs blinken uit waar zowel de input als output woorden zijn, en de regels vaag zijn.
  • Maar ze kunnen toch ook afbeeldingen? Klopt, maar daarvoor gebruiken ze andere modellen. Wanneer je ChatGPT vraagt om een afbeelding te genereren, gebruikt het eigenlijk een apart model getraind op visuele data.
  • Om het simpel te houden, blijven we bij taalmodellen.

Limitations

LLMs struggle with precision, real-time data, and tasks requiring exact logic.

  • Hallucinations - making up facts confidently
  • Mathematical calculations and precise logic
  • No access to real-time information
  • Consistent personality or memory across sessions
  • Understanding of physical world constraints
  • Inability to learn from individual conversations
  • LLMs zijn geen rekenmachines of waarheids-machines.
  • Ze kennen geen feiten; ze voorspellen waarschijnlijke antwoorden gebaseerd op patronen in data.
  • LLMs zijn uitstekende verhalenvertellers, maar niet altijd betrouwbare waarheids-vertellers. Dit kan wennen zijn.
  • We zijn gewend dat computers precieze, juiste antwoorden gevenâ€”zoals rekenmachines op school. Als een rekenmachine een fout antwoord geeft, is het een bug!
  • Met LLMs moeten we voor het eerst accepteren dat een computer verkeerd of onzeker kan zijn.
  • Oudere LLMs (zoals GPT-2) zouden vol vertrouwen vragen zoals "Hoe laat is het?" kunnen beantwoorden met een willekeurige, incorrecte tijd.
  • LLMs begrijpen je vraag en genereren een syntactisch correct antwoord, maar het is mogelijk niet feitelijk accuraat.
  • Moderne modellen lijken deze beperkingen te hebben overwonnen, maar eigenlijk zijn veel ervan niet opgelost door betere modellen te maken, maar door system prompts en software rond het model.

Cost

Pay per Token: Smallest unit of information a model processes

  • AI uses compute - costs based on operations performed
  • Token = small word or part of a word
  • Larger models = more compute = higher cost
  • Multi-modality complexity (images = tiles after size reduction)
  • THIS GETS REALLY COMPLEX, QUICKLY
  • Best approach: Try, measure tokens, calculate
  • Dus hoe worden kosten berekend? Per Token.
  • AI gebruikt compute. Dit is de capaciteit die nodig is om bewerkingen uit te voeren. Ik noemde dat de meeste populaire modellen die we tegenwoordig gebruiken next token generators zijn.
  • Betaal per Token - kleinste eenheid informatie die een model verwerkt. Kleine woorden, of onderdeel van een woord. zoals "de barkeeper", zou 3 tokens kunnen zijn, zou 4 kunnen zijn. Interessant spul, ik daag je uit om hier dieper in te duiken.
  • Klinkt eenvoudig? Maar het wordt snel complex. Grotere modellen gebruiken meer compute, dus het verwerken van een token is duurder.
  • Hoe groot een token is varieert per model. Afhankelijk van modelarchitectuur en ontwerpkeuzes.
  • Dan is er nog multi-modaliteit. Als een token onderdeel is van een woord, hoeveel is dan een afbeelding??? Dit wordt gedaan door het op te delen in tiles, maar alleen na het verkleinen.
  • Er is ook reasoning. Waarbij meerdere verzoeken naar het model worden gedaan na elke stap.
  • Voor complexe use cases, bijna onmogelijk om te voorspellen.
  • Beste manier is empirisch. Proberen, tokens gebruikt controleren. Berekenen.

  • Dit is ook ingebouwd in de system prompts van moderne modellen. Wanneer je GPT 5 een vraag stelt die meerdere stappen vereist om te beantwoorden, kun je het zien werken door het probleem stap voor stap.
  • Flipped Interaction: Een ander geweldig voorbeeld van een prompting pattern is Flipped interaction. Bij het prompten ben je waarschijnlijk gewend om vragen te stellen en antwoorden te krijgen, toch?
  • Maar wat als je de vragen niet kent? Wat als je een antwoord nodig hebt in een domein waar je niets van weet?
  • In plaats van meteen naar een antwoord te springen, speelt het model terug zoals een consultant: "Wacht, ik heb eerst meer info nodig."
  • Zeer effectief in domeinen waar je niet veel van weet.
  • Dit is ook een van de patronen die is ingebouwd in de system prompts van grotere modellen. Heeft iemand van jullie ooit Deep Research geprobeerd? Bijvoorbeeld van ChatGPT? Je kunt het een vraag stellen en het zal 5-20 minuten besteden aan onderzoeken, redeneren en een antwoord samenstellen.
  • Als je het een korte vraag stelt, zal het bijna altijd antwoorden door jou terug vragen te stellen. Omdat deep research veel tijd kost van het model, en tijd voor jou, en veel meer tokens kost dan een gewone vraag. Dus het zou zonde zijn als het je vraag niet correct begreep, dus het zal proberen het duidelijk te krijgen.

Defensive Prompting

  • Een ander onderwerp waar ik jullie over wil informeren is defensieve prompt engineering. Net als gewone applicaties kunnen AI-enabled applicaties gebruikt worden door beoogde gebruikers en kwaadwillende aanvallers. Alles wat gebruikt kan worden, zal ook misbruikt worden. Er zullen altijd gebruikers zijn die het proberen te misbruiken. Het prompten van een AI opent een nieuw palet van risico's.

Risks

Three main types of prompt attacks to defend against:

  • Prompt Extraction
    Extracting the application's system prompt to replicate or exploit
  • Jailbreaking & Prompt Injection
    Getting the model to perform harmful or unintended actions
  • Information Extraction
    Getting the model to reveal training data or context information
  • Risico's waar je bewust van moet zijn wanneer je een applicatie bouwt die de kracht van AI benut. Er zijn 3 hoofdtypen prompt aanvallen waar je je tegen wilt verdedigen:
  • Prompt extraction - Het extraheren van de applicatie prompt, inclusief de system prompt, ofwel om de applicatie te repliceren of te misbruiken
  • Jailbreaking en prompt injection - Het model slechte dingen laten doen
  • Information extraction - Het model zijn trainingsdata of informatie gebruikt in zijn context laten onthullen

Possible Consequences

  • Remote Code or Tool Execution
    Unauthorized execution of code or system tools
  • Data Leaks
    Extraction of private system and user information
  • Social Harms
    Providing knowledge for dangerous or criminal activities
  • Misinformation
    Manipulation of model outputs to spread false information
  • Service Interruption and Subversion
    Unauthorized access, incorrect decisions, or service disruption
  • Brand Risk
    Toxic or inappropriate statements causing PR crises
  • Prompt aanvallen brengen meerdere risico's met zich mee voor applicaties; sommige zijn verwoestender dan andere. Hier zijn er slechts een paar van:
  • Remote code of tool uitvoering - Voor applicaties met toegang tot krachtige tools kunnen hackers ongeautoriseerde code of tools uitvoeren. Stel je voor dat je een systeem hebt waarme je middels natuurlijke taal je database kunt bevragen. Die zal op basis van jouw vraag een SQL query genereren en die laten uitvoeren. Stel nou dat je hiermee alle gevoelige data van je gebruikers onthult of ongeautoriseerde e-mails naar je klanten stuurt. Als een ander voorbeeld, laten we zeggen dat je AI gebruikt om je te helpen een onderzoeksexperiment uit te voeren, wat het genereren van experimentcode inhoudt en het uitvoeren van die code op je computer. Een aanvaller kan manieren vinden om het model kwaadaardige code te laten genereren om je systeem te compromitteren.
  • Data lekken - Slechte actoren kunnen privÃ© informatie over je systeem en je gebruikers extraheren.
  • Sociale schade - AI modellen helpen aanvallers kennis en tutorials te krijgen over gevaarlijke of criminele activiteiten, zoals het maken van wapens, belasting ontduiken, en persoonlijke informatie exfiltreren.
  • Desinformatie - Aanvallers zouden modellen kunnen manipuleren om desinformatie uit te voeren ter ondersteuning van hun agenda.
  • Service onderbreking en ondermijning - Dit omvat het geven van toegang aan een gebruiker die geen toegang zou moeten hebben, het geven van hoge scores aan slechte inzendingen, of het afwijzen van een leningaanvraag die goedgekeurd had moeten worden. Een kwaadaardige instructie die het model vraagt om alle vragen te weigeren te beantwoorden kan service onderbreking veroorzaken.
  • Merk risico - Het hebben van politiek incorrecte en toxische uitspraken naast je logo kan een PR crisis veroorzaken, zoals toen Google AI search gebruikers aanspoorde om rotsen te eten of toen Microsoft's chatbot Tay racistische opmerkingen spuugde. Hoewel mensen misschien begrijpen dat het niet je bedoeling is om je applicatie beledigend te maken, kunnen ze de beledigingen nog steeds toeschrijven aan je gebrek aan zorg voor veiligheid of gewoon incompetentie.
